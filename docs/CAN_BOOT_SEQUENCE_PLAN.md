# План перехода на CAN-загрузку конфигов

## Цель

Front32 становится головным устройством CAN-сети. Он загружает свой
`config.json`, читает список `nodes`, загружает каждый файл `/node_<NAME>.json`,
ждет появления нод в BOOT и отправляет каждой ноде ее полный конфиг.

## Группы нод

Кроме персонального CAN ID у ноды может быть групповой ID, как дополнительный.
Группа задается в каждом файле ноды верхним параметром `group`, например:

```json
"group": "0x220"
```

Все ноды с одинаковым `group` должны слушать этот CAN ID как групповой адрес.
Групповая команда отправляется один раз на group ID, а все ноды этой группы
получают ее одновременно и синхронно исполняют, если команда им понятна.

Если команда ноде не относится или нода ее не понимает, на текущем этапе нода
просто игнорирует команду. Отдельный отчет об ignored/unsupported command можно
добавить позже, если он понадобится для диагностики.

## Этап 1. Загрузка конфигурации Front32

Сделано:

- Загружается `settings.UPDATE_URL`, но старая OTA-логика пока не меняется.
- Загружаются `CAN.tx`, `CAN.rx`, `CAN.bitrate`.
- Загружаются таймауты `CAN.timeouts_ms.ack` и `CAN.timeouts_ms.check`.
- Загружается массив `nodes`.
- Для каждой ноды из `nodes` требуется файл `/node_<NAME>.json`.
- пример конфига для каждой ноды можно посмотреть тут, это нода для управления столом  - json\node_TABLE.json.
- Полный JSON ноды сохраняется как payload для будущей отправки по CAN.
- Проверяются дубли имен нод, CAN ID и реальных MAC.
- Для головного CAN-конфига отключена старая проверка полного монолитного
  `device` через `MachineSpec::A`.

  если уже реализована загрузка параметра CAN.enabled то надо удалить, она не нужна. 

Важно:

- `mac: "00:00:00:00"` сейчас считается временной заглушкой и не мешает
  загрузке конфигов. Реальные MAC понадобятся на этапе BOOT discovery.
- Для синхронного движения PAPER+THROW обе ноды должны иметь одинаковый `group`.

## Этап 2. Разделение MachineSpec

- Разделить спецификацию на два уровня: головное устройство и нода.
- Спецификация головного устройства проверяет только локальное железо Front32 и
  логические требования к списку нод.
- Конфиг каждой ноды проверять так: 
  1. Убедится что это json
  2. "machine" = "NODE"
  3. "name" - уникален и соответствует одной из списка nodes головного.
  4. "group": "0x000" - либо нулевой, либо если группа укаазана надо найти соответсвие среди групп в разделе "groups"
  5. "mac":"00:00:00:00" - мак-адрес при загрузке автоматом проставляется в это поле. 
  6. "canID":"0x202" - уникальный CAN ID. 
- Старую монолитную `MachineSpec::A` не оставлять. надо переделать под новые требования.
Ее смысл теперь сводится к проверке указанных нод под тип станка.

## Этап 3. BOOT discovery

- Запустить CAN после загрузки config.json, используя параметры из секции CAN.
- Войти в окно ожидания всех нод.
- Слушать `Mgmt::BootHello`.
- Найти ноду по MAC из `BootHello`.
- Если MAC неизвестен, записать понятную диагностическую ошибку.
- Если MAC известен, но конфиг ноды отсутствует или невалиден, остановить BOOT.
- Назначить ноде ее CAN ID из конфига.
- Дождаться `AssignAck`.

## Этап 4. Передача конфига в ноду

- Отправить `ConfigBegin` с размером payload и CRC.
- Отправить JSON кусками через `ConfigChunk`.
- Завершить передачу через `ConfigEnd`.
- На каждом шаге ждать подтверждение или ошибку.
- При временных ошибках сделать ограниченное число повторов.
- При `ConfigError` остановить BOOT с деталями: нода, команда, код ошибки.
- Отправить `ApplyConfig`.
- Дождаться `ApplyResult`.

## Этап 5. Self-test и готовность

- Если на головном включен `CHECK_SYSTEM`, запустить self-test нужных нод. для этого отдельно есть состояние CHECK в которое головное после BOOT перейдет 
- Для каждой ноды дождаться `SelfTestResult`.
- После успешного применения конфига и self-test дождаться `ReadyReport`.
- Перевести систему в `IDLE` только когда все ноды готовы.

## Что нужно доработать в Can32

- Принять решение по CAN ID в Mgmt-протоколе: 8-битный короткий адрес замениить на полный
  11-битный standard CAN identifier.`Mgmt::AssignCanID`   надо расширить с `uint8_t` до `uint16_t`.
- Добавить helper для отправки полного JSON payload с повторами и проверкой CRC.
- Добавить поддержку группового адреса: нода должна слушать свой `canID` и
  свой `group`.

## Что нужно доработать в Back32

- Стартовать с минимальным дефолтным CAN-конфигом.
- Отправлять `BootHello` до назначения CAN ID.
- Принимать полный JSON config payload.
- Разбирать секцию `device` и настраивать локальные устройства.
- Подписываться на индивидуальный `canID` и групповой `group`.
- Поднимать нужные scenario servers по роли ноды.
- Выполнять role-specific self-test.
- Отправлять `ReadyReport` после успешной настройки.
