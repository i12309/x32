# AGENT.md for `harvester`

Этот файл фиксирует архитектурные правила проекта и рабочие договоренности.

## 1. Назначение проекта

Прошивка для ESP32 (PlatformIO + Arduino) для полиграфического станка (резка/связанные операции) с интерфейсом Nextion.

Ключевые части:
- управление моторами (`PAPER`, `TABLE`, `GUILLOTINE`, `THROW`) через FastAccelStepper;
- работа с датчиками/энкодером/муфтой через GPIO и MCP23017;
- конечный автомат состояний + планировщик шагов;
- UI на Nextion;
- HTTP/MQTT/OTA;
- хранение `config/data/stats` в LittleFS, служебные данные в NVS.

## 2. Точка входа

Файл: `src/main.cpp`

- `setup()`:
  - `Log::init()`
  - `State::init()` (старт с `BOOT`)
  - `Trigger::init()`
- `loop()`:
  - `Page::process()`
  - `State::process()`
  - `Service::process()`

## 3. Главные подсистемы

### 3.1 Core/Config/Data

Файлы: `src/Core.h`, `src/config.*`, `src/Data.*`

- `Core::config` работает с `config.json`.
- `Core::data` работает с `data.json`.
- `Core::settings` парсится из `config.settings`.
- `Data` хранит runtime-параметры процесса (`param`, `tuning`, `profiles`, `tasks`, `work`).

### 3.2 Состояния и план

Файлы: `src/State/State.h`, `src/State/PlanManager.*`

- Один активный `State*`.
- Переходы через `Factory(...)`.
- `PlanManager` хранит шаги (состояния/действия/параметры).
- `ACTION` исполняет callback.
- `DONE` возвращает управление в вызывающий контекст.

### 3.3 Загрузка (BOOT)

Файл: `src/State/Device/Boot.h`

BOOT выполняет: Nextion -> LittleFS -> NVS -> config -> Wi‑Fi -> OTA -> HTTP -> MQTT -> data/tuning/tasks/profiles -> Stats -> Registry -> trigger registration.

### 3.4 Реестр железа

Файл: `src/Controller/Registry.h`

`Registry` динамически создает устройства по `config.device`.
`Registry::Context` хранит общие ссылки на устройства для состояний.

### 3.5 Триггеры

- `Trigger` (`src/Controller/Trigger.*`) — polling.
- `McpTrigger` (`src/Controller/McpTrigger.*`) — IRQ от MCP.

Тяжелая логика только вне ISR.

### 3.6 Scene

Файл: `src/State/Scene.*`

`Scene` — единый фасад рабочих действий механизмов.

Правило: в новых состояниях и UI сначала использовать `Scene`, не дублировать low-level управление моторами.

## 4. Multi-Machine Rules (обязательно)

Проект работает на разных типах станков.

- Тип станка и его возможности определяются только из `config`.
- Нельзя жестко зашивать возможности конкретного станка без опоры на `config`.
- Логика должна учитывать фактически доступные механизмы текущего типа.

### Семантика папок `src/State`

- `A`, `B`, `C`, ...: бизнес-состояния только для конкретного типа станка.
- `AB`, `ACD`, ...: бизнес-состояния, общие только для перечисленных типов.
- `Device`: общие для всех типов состояния инициализации/проверки железа, без бизнес-логики производства.
- `Main`: простые универсальные состояния-кирпичики (например, `DetectPaper`), используемые в планах супер-состояний.
- `System`: системные состояния-классификаторы режима/функции, почти без бизнес-логики.

### Правило размещения новых состояний

1. Сначала определить, для каких типов станков состояние применимо.
2. Размещать состояние в соответствующей папке (`A`, `AB`, `Main`, `Device`, `System`).
3. Бизнес-логику резки/биговки размещать только в типовых папках (`A`, `AB`, ...).

## 5. Сервисы

- HTTP: `src/Service/HServer.*`
- MQTT: `src/Service/MQTTc.cpp`
- Сервисный цикл: `src/Service/Service.h`

`Service::process()` выполняется только в безопасных состояниях (`BOOT`, `IDLE`, `ERROR`, `SERVICE`, `NULL_STATE`).

## 6. Хранилище

- LittleFS: `/config.json`, `/data.json`, `/stats.json`
- NVS: `boot`, `wifi`, `licence`

## 7. Сборка

Файл: `platformio.ini`

- `env:esp32dev`
- `framework = arduino`
- `board_build.cxx_std = 17`
- post scripts: `post_build_script.py`, `post_build_ui_script.py`

## 8. Критичные правила изменений

1. Не делать I2C-операции в ISR.
2. Для движения использовать `IStepper`/`Scene`, чтобы не ломать статистику и консистентность.
3. Для новых сценариев использовать `PlanManager` (`ACTION`, `DONE`).
4. При изменении схемы `config` синхронизировать дефолты, загрузку и UI.
5. Учитывать offline MCP: часть устройств может быть недоступна, прошивка должна оставаться рабочей.

## 9. Чеклист перед завершением задач

- сборка проходит для `env:esp32dev`;
- Boot-переходы корректны;
- не сломаны `Registry::init()` и `Registry::Context::load()`;
- не сломаны `Scene::*Stop()`;
- проверены триггеры (`Trigger` и `McpTrigger`);
- проверены чтение/сохранение `/config` и `/data` через HTTP.
