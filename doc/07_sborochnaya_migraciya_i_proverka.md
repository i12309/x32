# Этап 7. Сборочная миграция и проверка

## Задача этапа

Добиться сборки головного режима под JC8048W550C/ESP32-S3 и постепенно отключить старые зависимости.

## Новый env PlatformIO

Старый env можно оставить как reference, но целевой нужен отдельный:

```ini
[env:jc8048w550c]
platform = espressif32
board = JC8048W550C
framework = arduino
board_build.cxx_std = 17
board_build.filesystem = littlefs
lib_extra_dirs =
  lib/ScreenUI/vendor
build_flags =
  -I src
  -D X32_TARGET_HEAD_UNIT=1
  -D X32_SCREEN_EEZ_LVGL=1
```

Board-файл `boards/JC8048W550C.json`, драйверы `esp32-smartdisplay`, локальный `lv_conf.h`, EEZ project, generated UI и Python-генераторы должны лежать внутри `x32`.

Целевая сборка не должна зависеть от соседних каталогов `C:\Users\sign\CODE\32\ScreenUI`, `C:\Users\sign\CODE\32\Screen32` или `C:\Users\sign\CODE\32\screenLIB`.

Пины CAN, дисплея, touch, подсветки и SD надо брать из локального board/config файла и дополнительно сверить по реальной плате JC8048W550C.

## Что копировать из соседних проектов

Из `ScreenUI`:

- `eez_project/SMIT_v1.eez-project`;
- `eez_project/src/ui/*`;
- `generated/shared/*`;
- `generated/frontend_meta/*`;
- минимальный набор `tools/ui_meta_gen/*`;
- только тот adapter-код, который нужен `Panel/EezPanel`.

Из `Screen32`:

- `boards/JC8048W550C.json`;
- `include/lv_conf.h` как основу;
- нужные части `lib/esp32-smartdisplay`;
- фрагменты платформенной инициализации дисплея/touch как reference для `DisplayDriver` и `TouchDriver`.

Из `screenLIB`:

- ничего не копировать по умолчанию;
- брать отдельные маленькие идеи или типы только если без них нельзя;
- не переносить protocol/client/host runtime, если экран работает внутри того же ESP32 и может вызывать C++ страницы напрямую.

После копирования нужно проверить, что PlatformIO собирает проект без внешних абсолютных include path.

## Обязательные комментарии в коде

Новый код для этой переделки нужно подробно комментировать на русском языке.

Обязательные правила:

- каждый класс должен иметь комментарий, зачем он нужен и какую ответственность закрывает;
- каждый публичный метод должен иметь комментарий, что делает метод, какие входные данные ожидает и какой результат возвращает;
- каждый protected/private метод тоже должен иметь комментарий, если его смысл не очевиден из имени;
- если внутри метода есть сложная логика, ветвление состояний, работа с CAN, ожидание task, retry, timeout, обработка ошибок или связь с EEZ/LVGL, рядом должен быть комментарий, что именно происходит и почему;
- комментарии должны объяснять смысл и причину решения, а не пересказывать очевидную строку кода;
- для временных заглушек обязательно писать, что это заглушка, почему она нужна и при каком этапе ее надо убрать.

## Простота реализации

Новый код должен быть написан простым C++.

Для первого рабочего варианта лучше выбрать понятное решение, даже если оно немного топорное:

- простые классы без глубокой иерархии;
- обычные `struct` для данных;
- явные методы вместо универсального runtime;
- обычный `switch` вместо сложных callback/registry цепочек;
- минимум шаблонов и C++-трюков;
- минимум скрытых зависимостей.

Если простой вариант выглядит временным, рядом должен быть `TODO` с объяснением, какой рефакторинг можно сделать позже. Нельзя усложнять первый вариант только ради универсальности.

Пример желаемого стиля:

```cpp
// Сцена подачи бумаги. Не управляет мотором напрямую:
// формирует задание для device, который внутри сам работает с мотором,
// датчиками и энкодером.
class PaperScene : public BaseScene {
public:
    // Запускает подачу бумаги на заданное расстояние.
    // Возвращает id scene-task, по которому состояние будет ждать Done/Failed/Timeout.
    SceneTaskId feed(float mm);
};
```

## Что должно собираться первым

1. Пустой `ICanBus`.
2. `Esp32TwaiBus` с begin/send/receive без интеграции в state.
3. `DeviceRegistry` без реальных device.
4. `Scene`, которая может вернуть failed/offline без падения.
5. `BOOT`, который проходит до диагностики отсутствующих device.
6. `Screen/Panel` с EEZ/LVGL и публичной страницей `Screen::load`.

## Заглушки

Так как Nextion backend не нужен, заглушки должны быть не для Nextion, а для незавершенных частей нового режима:

- fake `ICanBus` для сборки/тестов без железа;
- offline `DeviceNode`;
- методы объектной модели `Screen`, которые логируют несвязанную модель/элемент;
- объектная модель Screen, которая скрывает EEZ/generated ids от page-классов;
- soft `MachineSpec`, который пока не блокирует boot.

## Проверки без железа

- проект собирается;
- config читается;
- BOOT показывает загрузку;
- CAN init может быть отключен или mock;
- required device отмечаются offline;
- state machine не падает;
- ошибка понятна в логе/Screen.

## Проверки с CAN

1. `Esp32TwaiBus` стартует.
2. Видны heartbeat от device.
3. `DeviceRegistry` отмечает online/offline.
4. BOOT отправляет `configure`.
5. BOOT получает `ready`.
6. `CHECK` отправляет `check` всем required device.
7. Один простой scene task доходит до device и возвращает `Done`.
8. Отключение device дает offline/error.
9. Broadcast `stop` работает.

## Что убрать из головной сборки в конце

- прямой `Nextion.h` из экранных страниц;
- `lib/Nextion`;
- `NexUpdate`, `NexUploadHTTP`;
- прямое локальное управление `FastAccelStepper`;
- `MCP`, `I2CBus`, `McpTrigger`, если на головном ESP32 они больше не используются;
- жесткую валидацию старой `device.motors/sensors`.

## Критерий готовности

- `pio run -e jc8048w550c` собирается.
- BOOT остается состоянием и проходит через `oneRun/run`.
- `Screen/Panel` работает через EEZ/LVGL.
- `Screen/Page` сохраняет группировку старых UI-папок, но новые страницы называются без `p`.
- Публичный доступ к страницам короткий: `Screen::load`, `Screen::main`, `Screen::info`.
- Page-классы не используют id элементов.
- CAN-device видны в `DeviceRegistry`.
- `CHECK` и одно рабочее движение выполнены через `Scene`.
- Головное ESP32 не знает внутренний набор моторов и датчиков соседних device.
