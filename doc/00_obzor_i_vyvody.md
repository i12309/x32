# Глобальная переделка: обзор и выводы

## Цель

Переделать проект из локального контроллера станка в головное устройство на ESP32-S3 внутри экрана JC8048W550C.

Новая роль головного ESP32:

- показывать экранный интерфейс;
- хранить конфиг, профили, задания и статистику;
- выполнять state machine верхнего уровня;
- через `Scene` отправлять задания соседним CAN-device;
- получать статусы, ошибки и события;
- контролировать процесс, а не управлять моторами и датчиками в реальном времени.

## Главный вывод

`State` и `Scene` не удаляются.

Меняется только смысл нижнего слоя:

```text
Было:
  State -> Scene -> локальные моторы/датчики/GPIO/I2C

Станет:
  State -> Scene -> CAN device task -> status/done/error
```

`State` сохраняют структуру:

- `oneRun()` - один раз отправить задание или группу заданий через `Scene`;
- `run()` - ждать завершения, статуса или ошибки.

`Scene` сохраняется как уровень сценариев. Но вместо прямых вызовов `IStepper`, `ISensor`, `MCP` она будет формировать задания для одного или нескольких device на CAN.

## Важное ограничение

Головное ESP32 не должно знать внутренний набор моторов, датчиков, энкодеров и пинов конечного устройства.

Для головного устройства сосед по CAN - это абстрактный device:

- адрес;
- тип;
- протокол;
- имя;
- настройки, которые надо отправить при boot/configure;
- текущее состояние;
- список активных заданий.

Что находится внутри device, решает прошивка этого device. Например, один STM32-device может внутри иметь 4 мотора, 12 датчиков и энкодер, но головное устройство должно дать ему задание вроде `paper.feed_until_mark`, а не управлять каждым входом отдельно.

## Новая верхнеуровневая схема

```text
main.cpp
  App
    State machine
      Boot
      Check
      Process
      ...
    Scene
      send task to one device
      send task to many devices
      wait/collect task statuses
    DeviceRegistry
      abstract CAN neighbors
    CAN
      ICanBus
        Esp32TwaiBus
      MksCan
      StmCan
    Screen
      Page
        бизнес-логика страниц, с прежней группировкой папок
        имена без префикса p: load, main, info
      Panel
        инициализация, загрузка и обслуживание экрана
        объектная модель страницы поверх EEZ/LVGL
```

## Что сохранить

- `main.cpp`: `app.init()` и `app.process()`.
- `App`: точка сборки зависимостей.
- `State`, `oneRun()`, `run()`, `Factory()`.
- `PlanManager`, если он продолжит удобно задавать цепочки состояний.
- `Scene`, но с новой реализацией отправки заданий.
- `BOOT` как состояние state machine.
- Страницы как бизнес-классы внутри `Screen/Page`, но с новыми именами без префикса `p`: `pLoad` -> `load`, `pMain` -> `main`, `pINFO` -> `info`.

## Что изменить

- `Registry` больше не должен быть реестром локальных моторов/датчиков для головной сборки.
- Вместо `MotorProxy`, `SensorProxy` и других детальных фасадов нужен более общий `DeviceRegistry`.
- `MachineSpec` пока не должен блокировать разработку. На первом этапе его можно сделать пустым или максимально мягким.
- Экранный код лучше вынести из `src/UI` в новый модуль `src/Screen`.
- `Screen/Page` должен уйти от прямой зависимости `Nextion.h`.
- `Screen/Panel` должен управлять загрузкой EEZ/LVGL, дисплеем, touch и общим loop экрана.
- `Screen/Page` должен сохранить текущую группировку папок страниц.
- `Screen/Page` не должен использовать id элементов. Работа должна идти через именованную объектную модель страницы.
- Nextion-переходный backend не нужен. Нужен `Screen/Panel` на EEZ/LVGL и сборочные заглушки там, где страницы еще не перенесены.
- Конфиг должен описывать CAN, соседние device и настройки для них, а не GPIO всех исполнительных механизмов.

## Самодостаточность проекта

`x32` должен стать самодостаточным проектом.

Соседние проекты `ScreenUI`, `Screen32` и `screenLIB` можно использовать как доноры кода и как reference, но целевая сборка `x32` не должна зависеть от файлов по абсолютным путям вроде `C:\Users\sign\CODE\32\...`.

Что нужно перенести внутрь `x32`:

- EEZ Studio проект (`.eez-project`) как исходник интерфейса;
- сгенерированный EEZ/LVGL UI-код;
- сгенерированные `page_ids` / `element_ids` / object-map файлы;
- Python-генераторы, которые действительно нужны для обновления generated-файлов;
- драйверы дисплея и touch, нужные для JC8048W550C;
- board/config файлы PlatformIO для этой платы;
- локальные настройки LVGL.

При переносе не надо копировать целиком лишний runtime, demo, web-сборку, transport protocol и тестовые части, если они не нужны головному устройству. Если для первого запуска проще временно скопировать больше, такой код должен быть помечен `TODO`, а в комментарии должно быть написано, какую часть потом удалить или упростить.

## Стиль нового кода

Новый код этой миграции должен быть простым и понятным.

Правило важнее красивой архитектуры: сначала рабочий и читаемый код, потом рефакторинг.

- не использовать сложные возможности C++, если можно написать обычным классом, простым `struct`, `enum`, `switch` и понятными методами;
- не делать глубокую иерархию классов без необходимости;
- не прятать простую логику за шаблонами, type traits, callback-цепочками и универсальными абстракциями;
- не переносить сложность из `Screen32` или `screenLIB`, если в `x32` достаточно более простого локального фасада;
- если приходится сделать временно топорно, это нормально, но рядом нужен `TODO`, зачем так сделано и когда это можно привести в порядок;
- имена классов и методов должны быть ближе к текущему стилю проекта, чтобы код было легко читать вместе со старым `src/UI`, `State`, `App`.

## Рабочая стратегия

Переделку лучше вести так:

1. Сначала добавить общий CAN/device слой.
2. Затем переделать `Scene`, чтобы она умела отправлять задания device.
3. Потом по одному упрощать состояния: `oneRun()` отправляет задание, `run()` ждет результат.
4. После этого переносить экранный код в `Screen/Page` и `Screen/Panel`.
5. В конце выкидывать из головной сборки локальное управление моторами, MCP, Nextion и лишние legacy-зависимости.

## Правило комментариев

Весь новый код этой переделки должен подробно комментироваться на русском языке.

Обязательно комментировать:

- каждый класс;
- каждый метод;
- сложные участки внутри методов;
- CAN-задачи, ожидания, таймауты, retry, обработку ошибок;
- связь `State -> Scene -> DeviceRegistry`;
- связь `Screen/Page -> Screen/Panel -> EEZ/LVGL`;
- временные заглушки и условия их удаления.

Комментарий должен объяснять назначение и смысл, а не дублировать очевидную строку кода.
