# Этап 8. Экран и динамические Page

## Задача этапа

Сконцентрироваться на запуске экрана JC8048W550C и новом слое `Screen/Page`.

CAN на этом этапе не является целью. Он остается прототипным слоем, который должен собираться и не мешать boot/UI, но без требований к реальным соседним device, heartbeat и выполнению задач на железе.

Главный результат этапа: прошивка `jc8048w550c` показывает рабочий экран, проходит boot до понятного состояния и умеет переключать объектные страницы без старого Nextion UI.

## Главные ограничения

- Не возвращать прямую зависимость новых `Screen/Page` от `Nextion.h`.
- Не привязывать `Panel` к фиксированному enum-списку страниц.
- Не делать CAN обязательным для первого UI-запуска.
- Не тянуть старый `UI/` runtime в `jc8048w550c`.
- Не восстанавливать локальное управление моторами, MCP и `FastAccelStepper` в head-unit сборке.
- Новый экранный код писать просто: объектные страницы, явные методы, минимум универсального runtime.

## Целевое поведение первого запуска

1. Прошивка `pio run -e jc8048w550c` собирается.
2. После прошивки экран не черный.
3. На экране видна страница загрузки.
4. `BOOT` пишет текущий шаг загрузки на экран.
5. Если CAN/device отсутствуют, приложение не зависает без объяснения.
6. Ошибка или offline-режим отображаются на экране понятным текстом.
7. `Panel::process()` регулярно обслуживает LVGL/EEZ.
8. Старый `esp32dev` env продолжает собираться.

## Архитектура Page

`Panel` должен быть владельцем экранного runtime и навигации между объектами страниц.

`Page` должен быть базовым классом для конкретных страниц:

- `Screen::load`;
- `Screen::main`;
- `Screen::info`;
- `Screen::error`;
- сервисные и task-страницы позже.

Страница не должна регистрироваться в общем enum. Если конкретной странице нужен generated id EEZ/LVGL, он должен жить внутри самой страницы или небольшого adapter-кода этой страницы.

Пример желаемой модели:

```cpp
class load : public Page {
public:
    static load& getInstance();

    void show() override;
};
```

`Panel` должен знать только текущий объект:

```cpp
Page* activePageObject();
void show(Page& page);
void back();
```

## Что надо перенести/подключить

Из `ScreenUI`:

- EEZ project как reference;
- generated UI для минимального набора экранов;
- shared/frontend metadata, если они нужны для generated-кода;
- минимальные callbacks/actions, без лишнего runtime.

Из `Screen32`:

- `boards/JC8048W550C.json` уже перенесен;
- `lv_conf.h` как основу локальной конфигурации;
- нужную часть `esp32-smartdisplay` или минимальную инициализацию RGB display/touch;
- reference-код init дисплея и GT911 touch.

Не переносить:

- старый Nextion UI;
- web/demo runtime;
- protocol/client/host runtime из `screenLIB`, если он не нужен для локального экрана;
- универсальные registry/callback-системы, если достаточно прямых методов страниц.

## План работ

### 1. Подготовить локальные зависимости экрана

- Добавить локальный `lv_conf.h`.
- Добавить или подключить локальный `esp32-smartdisplay`.
- Проверить, что `jc8048w550c` видит include paths без абсолютных путей к соседним проектам.
- Сохранить `X32_SCREEN_EEZ_LVGL=1` как compile-time флаг экранного backend.

Критерий готовности: проект собирается с LVGL/display-зависимостями, даже если `Panel::init()` еще минимальный.

### 2. Реализовать минимальный `Panel::init()`

`Panel::init()` должен:

- инициализировать display;
- включить подсветку;
- инициализировать LVGL;
- инициализировать touch, если драйвер уже готов;
- инициализировать EEZ UI;
- загрузить стартовый экран или подготовить root UI.

Если touch временно не готов, оставить явный `TODO` с причиной и условием удаления заглушки.

Критерий готовности: после прошивки экран показывает хотя бы статическую LVGL/EEZ картинку.

### 3. Реализовать `Panel::process()`

`Panel::process()` должен регулярно обслуживать UI:

- вызывать LVGL handler;
- вызывать EEZ tick/update, если это требуется generated-кодом;
- не блокировать state machine.

Критерий готовности: экран обновляется, boot не зависает из-за UI loop.

### 4. Доделать страницу `Screen::load`

Минимально нужны:

- `load::show()`;

`BOOT` уже вызывает `loadPage().show()`, `setStatus()` и `setProgressColor()`, поэтому первая настоящая интеграция должна идти через `load`.

но от `setStatus()` и `setProgressColor()` надо отказаться потому что на Eez ui элементов уже нет. 

Критерий готовности: шаги boot видны на экране.

### 5. Доделать страницу `Screen::error`

Нужно, чтобы ошибки boot/UI были видны без Serial:

- `error::showError(title, message)`;
- отображение заголовка;
- отображение текста;
- желательно короткий код/категория ошибки.

Критерий готовности: при искусственной ошибке на экране появляется понятная error-страница.

### 6. Доделать страницу `Screen::info`

Нужна простая универсальная информационная страница:

- три текстовые строки;
- OK;
- Cancel опционально;
- callbacks должны жить на уровне page-класса, а не в `Panel`.

Критерий готовности: `Screen::info::showInfo()` открывает страницу и вызывает callbacks.

### 7. Сделать offline-режим без CAN-железа

Так как реальных CAN-device пока нет, boot не должен требовать их для проверки экрана.

Нужно выбрать простой временный механизм:

- настройка в config: CAN disabled / allow missing devices;
- или compile-time режим head-unit UI prototype;
- или пустой список required device для первого запуска.

Поведение должно быть явным:

- в лог пишется, что CAN/device слой работает в prototype/offline режиме;
- на экране можно показать `CAN prototype mode`;
- отсутствие CAN не считается ошибкой UI-запуска.

Критерий готовности: плата доходит до `IDLE` или `main` без реального CAN.

### 8. Доделать страницу `Screen::main`

Минимальная main-страница нужна после успешного boot:

- название/режим;
- состояние CAN prototype/offline;
- версия прошивки;
- короткий статус системы.

Критерий готовности: после boot устройство переходит с load на main.

### 9. Проверить touch

После стабильной картинки:

- проверить GT911 init;
- проверить координаты;
- проверить swap/mirror;
- добавить одну тестовую кнопку на `main` или `info`.

Если touch пока не работает, экран все равно должен запускаться и обновляться.

Критерий готовности: хотя бы одна кнопка нажимается и вызывает действие.

### 10. Вернуть HTTP позже, отдельно

`Service/HServer.cpp` сейчас нельзя возвращать в head-unit без чистки, потому что он тянет старый `UI/Main/pWAIT.h`.

После запуска экрана:

- убрать зависимость `HServer` от старого UI;
- заменить UI-вызовы на `Screen::info/error` или callback-интерфейс;
- вернуть `Service/HServer.cpp` в `build_src_filter`;
- включить `StartHttp` в `BOOT` для head-unit.

Это отдельный подэтап, не блокирующий первый запуск экрана.

## Минимальный порядок реализации

1. LVGL/display init.
2. Статическая картинка на экране.
3. `Panel::process()`.
4. `Screen::load`.
5. Offline boot без CAN.
6. `Screen::error`.
7. `Screen::main`.
8. Touch.
9. `Screen::info`.
10. Возврат HTTP без старого UI.

## Проверки

Обязательные проверки после каждого крупного шага:

```bash
pio run -e jc8048w550c
```

esp32dev - надо будет убрать

Проверки на железе:

- экран включил подсветку;
- нет белого/черного пустого экрана после boot;
- Serial показывает шаги boot;
- на экране видны шаги boot;
- без CAN плата не зависает;
- ошибка отображается на экране;
- touch не ломает обновление экрана.

## Критерий завершения этапа

Этап можно считать завершенным, когда `jc8048w550c` запускается на реальной плате, показывает `load`, проходит boot в CAN prototype/offline режиме, переходит на `main` или показывает понятную `error`, а страницы управляются объектной моделью без `PageId` и без старого Nextion UI.
